**Filed:** 2026-06-02 20:00 local (18:00Z)
**Observed:** 2026-06-02, ~19:40–19:50 local (~17:40–17:50Z)
**Status:** root cause not found in software; self-cleared; leading suspicion is a transient hardware (memory) fault
**Severity:** non-blocking; single occurrence; not since reproduced
**Evidence archive:** [`anomaly-evidence-2026-06-02/`](anomaly-evidence-2026-06-02/) (binaries, disasm, diff, meta — see its `MANIFEST.md`)

## One-paragraph summary

While validating a record-value `Exit` fix, the same `compiler/pascal26`
binary (md5 unchanged across the whole episode) intermittently produced a
*different*, crashing executable from the same source file. The divergence was a
handful of bytes at identical total file size, clustered as a consistent address
offset — the shape of a single corrupted in-memory value propagated to several
emit sites, not scattered random damage. It could not be reproduced afterward in
400 consecutive compiles. The toolchain is deterministic by construction, so the
leading hypothesis is a transient memory bit flip on this machine's non-ECC RAM.

## What happened (timeline)

1. Fixed a real bug: returning a record value via `Exit(r)` lowered the source
   with `IRLowerAST` (which for an lvalue emits `IR_LOAD_SYM`, i.e. the first 8
   bytes loaded as a value); `IR_COPY_REC` then misread that value as the source
   address and segfaulted. Fix takes the source *address* for lvalue records.
   This fix is sound and independently verified.
2. `make` (self-host) converged; the trimmed regression passed standalone.
3. First `make test` after wiring the regression: the produced
   `test_managed_record_exit26` **segfaulted** (rc 139, core dumped).
4. Manual recompiles of the identical source with the identical compiler:
   **passed**, byte-stable.
5. Second `make test`: **segfaulted again**. Compiler md5 verified unchanged
   (`badd42…`).
6. `cmp` of a good binary vs. the crashing one: they **differed** (same size).
   Same compiler, same source, different output — the impossible-for-deterministic-software
   event.
7. Chased triggers and ruled them out (output-path length, output-path text,
   fresh-vs-overwrite, pre-existing larger target, empty/large/make-like
   environment) — all produced the good binary.
8. The divergence then **stopped on its own**. 400 consecutive recompiles (200
   compiler self-compiles + 200 of the failing test) were byte-identical with
   zero crashes; full `bootstrap` / `test` / `test-nilpy` green.

## Byte-level forensics (surviving evidence)

The bad binary was overwritten before it could be archived (the trigger-chase
recompiled over it). The one captured `cmp -l good bad` survives. Good = a
passing build; both binaries were 4009 bytes.

`cmp -l` output (1-based offset, octal old → octal new):

```
  97 251 201      105 161 211      125 251 201      205 261 211      223 271 221
  98  17  26      106  40  47      126  17  26      206  17  26      224  17  26
```

Decoded as little-endian 16-bit fields (file offset = listed offset − 1):

| File offset | Old (LE16) | New (LE16) | Delta |
|-------------|-----------|-----------|-------|
| 0x60 | 0x0FA9 | 0x1681 | **+0x06D8** |
| 0x68 | 0x2071 | 0x2789 | +0x0718 (= 0x06D8 + 0x40) |
| 0x7C | 0x0FA9 | 0x1681 | **+0x06D8** |
| 0xCC | 0x0FB1 | 0x1689 | **+0x06D8** |
| 0xDE | 0x0FB9 | 0x1691 | **+0x06D8** |

### Confirmed: the corrupted value is the output image size `0x0fa9`

Mapping the offsets to the ELF structure (Phdr at 0x40, entry at 0x78) and
disassembling a reconstruction (good binary + the 10 known byte changes) pins it
down. `0x0fa9` = 4009 = the real file size, and it is the value the compiler
emits in every differing location:

- **0x60 = `p_filesz`** of the single `LOAD` segment: `0x0fa9` → `0x1681`.
- **0x68 = `p_memsz`**: `0x2071` (= filesz + 4296 B BSS) → `0x2789` (+0x0718).
- **0x7C = the entry instruction's address operand.** At `0x400078` the good
  binary has `48 89 24 25 a9 0f 40 00` = `mov %rsp, 0x400fa9` — the runtime
  saving the stack pointer to `base + image-size`. Corrupt: operand → `0x401681`.
- **0xCC / 0xDE = two more absolute-address operands** referencing
  `base + size (+8 / +16)`.

So a single logical value — the output **image size** — was corrupted by
**+0x06D8** and propagated to `p_filesz`, `p_memsz` (as +0x0718, the derived
filesz+BSS), and three embedded `base+size` address operands. Two independently
fatal results:

1. `p_filesz = 0x1681` (5761) **> actual file size (4009)** → the kernel maps the
   `LOAD` segment past end-of-file → SIGSEGV at exec.
2. `mov %rsp, 0x401681` stores the stack pointer to the wrong address.

The **partial reconstruction (only the 10 known bytes) already segfaults
(rc 139)**, confirming mechanism (1). File size stayed 4009 → it was not a
layout/section growth, just a corrupted size *value*. The magnitude +0x06D8 is
multi-bit (not a power of two), so it was not a clean single-bit flip of the
final field; consistent with a flip in an upstream intermediate holding the size
(or a transient multi-bit event) — the precise upstream cannot be recovered.

Hashes recorded at the time:
- good binary: `a9f415e4e45eb8af29889b6f1a0ce548` (4009 B) — archived.
- original crashing binary: `d1e2421c1d44b71a647505d8b09dd010` (4009 B) —
  **overwritten during the trigger-chase; not recoverable.**

**Evidence gap:** the live `cmp -l` was piped through `head`, so only the first
10 differing bytes survive. The full diff is lost with the original binary; the
reconstruction therefore segfaults but is not byte-identical to the original
(md5 `d8c23d0b821e5aabaad83fcafab7993e`). All survivors are in
[`anomaly-evidence-2026-06-02/`](anomaly-evidence-2026-06-02/).

**No core dump exists:** `ulimit -c` was `0` and `core_pattern` pipes to apport,
which discards non-packaged binaries; the shell's "core dumped" notwithstanding,
nothing was persisted.

## Why software causes were ruled out (as far as possible)

- **No randomness, no timestamps, no per-compile threads.** The compiler's
  output is a pure function of its input by design.
- **Self-host gate enforces determinism:** `bootstrap`/`test` require
  `build == verify` byte-identical, and that passed before, during (md5 stable),
  and after.
- **Same compiler md5** produced both good and bad output — so it is not a code
  path that changed.
- **Trigger isolation failed:** path length, path text, overwrite vs. create,
  pre-existing target contents, and environment size all reproduced the *good*
  output only.
- **400× post-event canary clean** — if it were a latent uninitialized-read or
  data-dependent codegen bug, that volume on the exact triggering input would
  very likely have surfaced it again.

The combination — deterministic toolchain, single non-reproducible divergence,
correlated single-offset corruption, **non-ECC memory with no error logging** —
points away from a compiler defect and toward a transient hardware memory fault
(cosmic-ray / marginal-cell / thermal bit flip). It cannot be *proven* hardware
without ECC telemetry, which this machine lacks.

## What was NOT affected

- The `Exit(record)` source-address fix is correct on its own merits and is
  covered by `test/test_managed_record_exit.pas`.
- No data loss; the committed/pushed binaries and sources are byte-stable and
  reproduce cleanly.

## If it recurs — diagnostics

```sh
# Determinism canary: deterministic compile must be byte-stable.
./compiler/pascal26 test/hello.pas /tmp/golden
for i in $(seq 1 500); do
  ./compiler/pascal26 test/hello.pas /tmp/t
  cmp -s /tmp/golden /tmp/t || echo "DIVERGED run $i"
done
```

```sh
journalctl -k --no-pager | grep -iE 'mce|edac|hardware error|machine check|memory error'
dmesg | grep -iE 'edac|mce'
```

- Run **memtest86** (overnight) — the only reliable detector for non-ECC DRAM
  faults on this host.
- Watch CPU/DRAM temperatures and check for overclock / XMP instability.
- For long unattended self-host or release builds, prefer a machine with **ECC
  RAM** so silent flips become logged corrected/uncorrected events.
